# PR Summary 작성 원칙

1. PR의 목적을 가장 먼저 파악한다.
- 버그 수정인지, 리팩토링인지, 기능 추가인지 명확히 한다.

2. 단순 변경보다 목적 중심으로 서술한다.
- "코드 정리" → "불필요한 rerender를 제거하기 위해 컴포넌트 책임 분리"

3. diff 전체를 나열하지 않는다.
- 의미 단위(파일 단위, 기능 단위, 역할 단위)로 묶는다.

4. 변경의 영향 범위(Breaking changes 등)를 반드시 명시한다.

5. 중요한 구조 변경은 Before vs After로 설명한다.

6. PR Body가 부족하면 diff 기반으로 맥락을 추정하되, 단정하지 않는다.
- “추정컨대 ~ 개선을 위한 변경으로 보인다.”

7. 리뷰어가 빠르게 이해할 수 있게 핵심 내용을 앞부분에 압축하여 제시한다.

8. 코드 추가와 코드 삭제의 의도를 명확히 구분한다.
