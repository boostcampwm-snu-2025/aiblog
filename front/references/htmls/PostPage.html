Understanding Async/Await in JavaScript

Published on November 06, 2025

If you've spent any time with modern JavaScript, you've likely run into asynchronous code. Dealing with callbacks ("callback hell") and even complex `.then()` chains can be confusing. This is where `async/await` comes in.

What is Asynchronous Code?

By default, JavaScript is a single-threaded, synchronous language. This means it executes one line of code at a time. If one line takes a long time (like fetching data from an API), it blocks all other code from running.

Asynchronous operations (like `fetch`, `setTimeout`, or reading a file) allow the JavaScript engine to "pause" that task, work on other things, and come back to the task once it's complete.

"Async/await is syntactic sugar built on top of Promises, making asynchronous code easier to write and read. It makes async code look more like synchronous code."

How it Works

There are two key pieces to the puzzle:

  • `async` function: The `async` keyword is placed before a function declaration. This automatically makes the function return a Promise.
  • `await` operator: The `await` keyword is used inside an `async` function. It pauses the function's execution until a Promise is settled (either resolved or rejected).

Example: Fetching Data

Here's how you might fetch data using `async/await`.

const fetchUserData =
              async () => {
              try {
              const response = await fetch(
              'https://api.example.com/users/1'
              );

              if (!response.ok) {
              throw new Error('Network response was not ok');
              }

              const userData = await
              response.json();
              console.log(userData);

              } catch (error) {
              console.error(
              'Failed to fetch user:', error
              );
              }
              };

              fetchUserData();

Notice the `try...catch` block. This is the standard way to handle errors in `async/await` functions, just like you would in synchronous code. It's clean, readable, and powerful.
